从你的错误信息可以看出，**DBus 服务 `org.kde.kpmcore.helperinterface` 已经在运行（进程 PID 1127），但普通用户无权向它发送消息**。这正是你的程序卡在 `loop.exec()` 的根本原因：DBus 调用因为权限拒绝而永远得不到回复，导致事件循环挂起。

---

### 🔍 错误解读

```
Error org.freedesktop.DBus.Error.AccessDenied: Rejected send message, 1 matched rules; ...
destination="org.kde.kpmcore.helperinterface" (uid=0 pid=1127 comm="...")
```

- 目标服务运行在 root 用户（uid=0）下。
- DBus 有一条匹配规则（`1 matched rules`），但规则拒绝了你的请求。
- 你的程序是以普通用户（uid=1000）运行的，因此无权与该服务通信。

---

### 🛠️ 解决方案：调整 DBus 策略文件

DBus 系统总线的访问控制由 `/usr/share/dbus-1/system.d/` 中的 XML 策略文件决定。我们需要找到控制 `org.kde.kpmcore.helperinterface` 的策略文件，并修改它，允许普通用户调用该服务的方法。

#### 1. 查找对应的策略文件

```bash
grep -r "org.kde.kpmcore.helperinterface" /usr/share/dbus-1/system.d/
```

通常会输出类似：
```
/usr/share/dbus-1/system.d/org.kde.kpmcore.helperinterface.conf:  <policy user="root">
...
```

如果没有输出，也可能是文件名为 `kpmcore.conf` 或其他。可以用通配符搜索：

```bash
ls /usr/share/dbus-1/system.d/ | grep -i kpmcore
```

#### 2. 查看策略文件内容

假设找到的文件是 `/usr/share/dbus-1/system.d/org.kde.kpmcore.helperinterface.conf`，用 `cat` 查看：

```bash
cat /usr/share/dbus-1/system.d/org.kde.kpmcore.helperinterface.conf
```

典型内容可能如下：

```xml
<!DOCTYPE busconfig PUBLIC "-//freedesktop//DTD D-BUS Bus Configuration 1.0//EN"
 "http://www.freedesktop.org/standards/dbus/1.0/busconfig.dtd">
<busconfig>
  <policy user="root">
    <allow own="org.kde.kpmcore.helperinterface"/>
    <allow send_destination="org.kde.kpmcore.helperinterface"/>
    <allow receive_sender="org.kde.kpmcore.helperinterface"/>
  </policy>
  <policy context="default">
    <deny own="org.kde.kpmcore.helperinterface"/>
    <deny send_destination="org.kde.kpmcore.helperinterface"/>
  </policy>
</busconfig>
```

这里的关键是 `<policy context="default">` 中有一条 `deny send_destination="org.kde.kpmcore.helperinterface"`，它会拒绝任何未明确允许的用户发送消息到该服务。而 `<policy user="root">` 只允许 root 用户。

#### 3. 修改策略文件以允许普通用户

你需要在 `<policy context="default">` 前面添加一条针对你用户的策略，或者放宽默认策略。

**方法 A：为特定用户（如当前登录用户）添加允许规则**

```bash
sudo nano /usr/share/dbus-1/system.d/org.kde.kpmcore.helperinterface.conf
```

在 `<busconfig>` 内，`<policy user="root">` 之后，添加以下内容（假设你的用户名是 `cutefish-live`，UID=1000）：

```xml
  <policy user="cutefish-live">
    <allow send_destination="org.kde.kpmcore.helperinterface"/>
    <allow receive_sender="org.kde.kpmcore.helperinterface"/>
  </policy>
```

**方法 B：直接修改默认策略为允许（不安全，仅用于测试）**

将默认策略中的 `<deny>` 改为 `<allow>`，或者删除该段。但这样所有用户都能访问，风险较大，建议仅测试时使用。

#### 4. 重启 DBus 使配置生效

修改后，需要重启 DBus 服务（注意：这可能会暂时中断系统服务，请在安全环境下操作）：

```bash
sudo systemctl restart dbus
```

或者如果不想重启整个 DBus，可以只重新加载配置（通常需要重启服务）：

```bash
sudo pkill -HUP dbus-daemon   # 谨慎使用
```

更稳妥的方法是重启系统，或者至少重启相关的服务进程（例如重新启动 `kpmcore_external` 进程）。但最简单的是重启 DBus。

#### 5. 验证权限是否生效

再次以普通用户执行 DBus 调用：

```bash
dbus-send --system --print-reply --dest=org.kde.kpmcore.helperinterface /Helper org.freedesktop.DBus.Introspectable.Introspect
```

如果返回 XML 描述信息（而不是 AccessDenied），说明权限问题已解决。

#### 6. 重新运行 Calamares

现在再启动 Calamares，应该不会再卡在 `loop.exec()` 了。

---

### 📌 补充说明

- 如果你发现策略文件路径不是 `/usr/share/dbus-1/system.d/` 而是 `/etc/dbus-1/system.d/`，操作相同。
- 如果策略文件内容中没有 `<policy context="default">`，而是 `<policy group="...">` 之类的，可以参照添加用户策略。
- 如果你对修改系统文件不放心，可以先备份原文件。
- 此问题与 polkit 无关，是 DBus 层级的权限限制，因此之前的 polkit 规则无法解决。

如果修改后仍无法访问，请检查服务进程的 DBus 绑定是否正确（如是否监听在系统总线），并再次确认策略文件生效。
